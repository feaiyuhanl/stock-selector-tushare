## ADDED Requirements

### Requirement: 默认执行流程耗时阶段可区分

系统 SHALL 在文档中明确默认执行流程的耗时阶段划分（启动、数据新鲜度检查、选股数据加载、评分计算、复盘、飞书同步、通知），并说明各阶段的主要耗时来源，便于用户与开发者定位性能瓶颈。

#### Scenario: 文档包含耗时阶段说明

- **WHEN** 用户或开发者查阅性能分析文档
- **THEN** 文档中列出默认执行各阶段名称与主要耗时来源（如 API、I/O、CPU）

### Requirement: 必须与非必须步骤可区分

系统 SHALL 在文档中界定「必须」与「非必须」执行步骤：必须步骤为选股核心路径（Token 校验、股票池获取、数据加载与评分、保存推荐）；非必须步骤包括数据新鲜度报告、二次重试、自动复盘、飞书同步、邮件通知等。关闭或延后非必须步骤 SHALL 不影响「得到 TOP 推荐并落库」的结果正确性。

#### Scenario: 关闭非必须步骤后仍得到推荐结果

- **WHEN** 用户通过配置或 CLI 关闭数据新鲜度报告、自动复盘、飞书同步等非必须步骤
- **THEN** 选股仍能完成并保存推荐结果，且 TOP 推荐与全量执行时一致（在相同数据与参数下）

### Requirement: 主要阻塞点与优化方案可查阅

系统 SHALL 在文档中说明主要阻塞点与卡点（如阶段2 数据加载的 API 限频与单股串行、二次重试串行与延迟、数据新鲜度报告全量检查、复盘阶段大量 K 线查询、combined 模式串行执行），并列出可实施的优化方案与优先级（如可配置跳过数据新鲜度报告、二次重试并行/限流、默认线程数配置、复盘/飞书可关闭、combined 并行等），以及各方案的风险或注意点。

#### Scenario: 用户按文档实施优化

- **WHEN** 用户按性能分析文档中的优化方案与优先级进行配置或代码调整
- **THEN** 可在不破坏选股正确性的前提下缩短默认执行耗时或减少非必要步骤

### Requirement: 数据新鲜度报告可配置跳过（可选实现）

当实现该能力时，系统 SHALL 提供配置项或 CLI 参数（如 `--skip-freshness-report` 或 `SKIP_FRESHNESS_REPORT`），使选股前不执行数据新鲜度报告（`print_data_freshness_report`）。默认值 SHALL 为不跳过，以保持现有行为。

#### Scenario: 启用跳过时选股前不输出新鲜度报告

- **WHEN** 用户设置跳过数据新鲜度报告为 True（或传入对应 CLI 参数）
- **THEN** 选股流程开始前不调用 `print_data_freshness_report`，选股结果与未跳过时一致
